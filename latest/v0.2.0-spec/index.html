    v0.2.0 Web 控制台 — Spec（待用户拍板）¶

    状态：📋 草案 / 待你醒来 review 作者：Claude (auto-drafted) 日期：2026-06-12 凌晨

    Context¶

    接力卡与 .session-handoff.md 都把 v0.2.0 Web 控制台列为 🟢「可做但未做」项。 本次 session 用户已选「全部实现，一点一点来」，本 spec 是 v0.2.0 的第一阶段产物： brainstorm + 架构选型 + 实施分阶段。未写实现代码（避免在没有方向确认时乱动）。

    1. 需求 brainstorm¶

    代理是「无头」服务 —— 用户目前只能： - 看 stdout 日志 - 翻 SQLite 数据库 - 跑 curl /v1/health

    这意味着用户调试时是「盲飞」。Web 控制台的目标 = 可视化调试 + 实时监控。

    候选功能（影响 ROI 排序）¶

    # 功能 用户价值 实现成本 建议
    A 实时请求流（SSE tail / 最近 100 条） 🟢 极高 🟡 中 ✅ MUST
    B 缓存命中率面板（24h 折线 + 命中/未命中饼图） 🟢 极高 🟢 低 ✅ MUST
    C 配置热更新（不重启改 yaml 部分字段） 🟡 中 🟠 中高 ⏸ v0.2.1
    D 健康检查 + 上游连通性（一键测试 MiniMax-M3） 🟡 中 🟢 低 ✅ MUST
    E 工具调用日志（SSR tools 拦截 / 执行 / 回灌） 🟢 高 🟡 中 ✅ MUST
    F Token 消耗统计（按模型/时间/用户） 🟡 中 🟠 中高 ⏸ v0.2.1
    G 代理设置面板（改 yaml、持久化） 🟡 中 🟠 中高 ⏸ v0.2.1

    v0.2.0 范围 = A + B + D + E（4 个面板，一次性推出）

    2. 架构决策¶

    决策 1：技术栈¶

    选项 优势 劣势
    FastAPI Jinja2 templates + htmx 零构建、Python 一把梭、SSR 友好 UI 不够 fancy
    FastAPI + React SPA UI 灵活、组件化 需要 npm build 流程、复杂度↑
    FastAPI + Vue 3 CDN 折中 双技术栈心智负担

    推荐：FastAPI Jinja2 + htmx + Alpine.js + Pico CSS

    理由： - 14 个依赖已够多，不加 npm 工具链 - htmx 让 90% 的"局部刷新"用 5 行 HTML 搞定 - Pico CSS 1 个 CDN 链接搞定主题，不写 CSS - 控制台是「运维/调试」场景，不是「产品」场景，够用就好

    决策 2：数据流¶

    [FastAPI 现有 /v1/* 端点] ── hook ──► [内存环形 buffer] ── poll 1s ──► [Web 控制台 SSE]
                                        (deque(maxlen=1000))

[Cache Store 已有 sqlite] ── query ──► [/admin/api/cache/stats] ──► [htmx 折线图]

[Settings 已加载] ── POST /admin/api/config/reload ──► [SIGHUP 给自己]

    关键约束： - 不影响现有 /v1/* 行为（hook 写 buffer 用 try/except 包裹） - Web 控制台本身是只读（v0.2.0 不做配置写） - 认证：v0.2.0 默认绑定 127.0.0.1:8788（与代理不同端口），不暴露公网 - 未来 v0.2.1 加 basic auth 时不影响 API

    决策 3：端口与路径¶

    路径 用途
    :8787/v1/* 现有 Claude Code 兼容端点（不变）
    :8788/admin/ 新增 Web 控制台（独立端口，独立进程？嵌进现有 app）

    推荐：控制台路由挂到现有 app（不分离进程），用 127.0.0.1 绑定防御。理由：少一个 systemd unit，少一份部署心智。

    决策 4：可观测性数据源¶

    数据 来源 采集成本
    请求流 中间件 hook（request_started / request_finished） 🟢 加 1 个 loguru sink
    缓存命中率 CacheStore 已记录 hits/misses，加 1 个 counter 🟢 现成
    工具调用 ServerSideToolsHub.execute() 已有 log，加 1 个 hook 🟢 现成
    上游连通性 UpstreamClient 加 1 个 ping() 方法 🟢 现成

    所有数据都已存在或近似存在，v0.2.0 主要是「把它们暴露成 JSON / SSE」。

    3. 实施分阶段¶

    Phase 1：基础设施（半天）¶

    • requirements.txt 加 jinja2 + htmx（CDN 引用不计）
    • proxy/admin/ 新建子包
    • proxy/main.py 挂载 /admin/ 路由
    • Settings.server.admin_port = 8788 + admin_enabled = False 默认关
    • 配置：用户开 admin_enabled: true 才暴露 :8788

    Phase 2：API endpoints（半天）¶

    • GET /admin/api/requests/recent?limit=100 → JSON
    • GET /admin/api/requests/stream → SSE（实时 tail）
    • GET /admin/api/cache/stats?window=24h → JSON
    • GET /admin/api/tools/recent?limit=50 → JSON
    • GET /admin/api/upstream/ping → JSON（试发一个 mini 请求）

    Phase 3：UI 模板（一天）¶

    • proxy/admin/templates/base.html（Pico CSS + 暗色）
    • proxy/admin/templates/index.html（4 卡片入口）
    • proxy/admin/templates/requests.html（实时表格 + htmx）
    • proxy/admin/templates/cache.html（折线 + 饼图，cdn 拉 Chart.js）
    • proxy/admin/templates/tools.html（时间线）
    • proxy/admin/templates/upstream.html（Ping 按钮 + 结果）

    Phase 4：测试（半天）¶

    • tests/test_admin_api.py —— 8-10 个单元测试
    • tests/test_admin_e2e.py —— 4-5 个 E2E（用 mock）
    • CI 加 admin port 检查

    Phase 5：文档（半天）¶

    • docs/admin.md 用户文档
    • README 加 admin 段 + 截图
    • CHANGELOG 加 [0.2.0]
    • config.yaml.example 加 admin 配置示例

    总投入：~3 天（1 人），1 个 PR

    4. 风险与回退¶

    风险 缓解
    控制台 hook 拖慢主流程 所有 hook 用 try/except 包裹，失败仅记 log
    内存 buffer 泄漏 deque(maxlen=1000) 自动驱逐
    htmx 兼容性问题 备选：原生 fetch + 5 行 JS
    用户误开 admin_enabled: true 暴露公网 默认关 + README 红色警告
    中间件 hook 顺序冲突 在 main.py:188 之前注册，独立 include_router

    5. 不做清单（v0.2.0 明确不做）¶

    • ❌ 写配置（只读 + reload）→ v0.2.1
    • ❌ 用户认证 → v0.2.1（依赖路径复杂度）
    • ❌ Token 消耗统计 → v0.2.1（需要埋点，跨模块）
    • ❌ Docker 化 → 已降级到 v0.5.0+
    • ❌ Grafana / Prometheus 集成 → 过度设计

    6. 待你醒来的决策¶

    1. 范围确认：A+B+D+E 4 个面板是否同意？还是想加/减？
    2. 技术栈：Jinja2 + htmx + Pico CSS 行吗？还是想要 React/Vue？
    3. 端口：8788 行吗？还是想用同一端口加路径？
    4. 认证：v0.2.0 真的不认证吗（只绑 127.0.0.1）？
    5. 优先级：Phase 1+2+3+4+5 一次推？还是先 Phase 1+2（API only）看效果再推 UI？
